## Challenges Encountered

### Challenge 1: Understanding Premiere UXP's Action-Based API
**Description:** The most significant challenge was understanding and correctly implementing Premiere's Action-based API for modifications like creating a bin. It's not a direct function call but requires creating an `Action` object and executing it within a `transaction`.  
**Resolution:** ✅ This was resolved by carefully reading the official Adobe UXP documentation for `FolderItem` and `Project` classes and experimenting with the `createBinAction` and `executeTransaction` methods.  
**Impact:** This discovery is crucial for all future development on this plugin, as it establishes the correct pattern for any project modifications.

---

## Key Learnings & Insights

1.  **Action-Based, Transactional Model:** Premiere Pro's UXP API relies heavily on an Action-based, transactional model for any project modifications. This is a fundamental concept for any future development on this plugin.
2.  **Dual-Mode Architecture:** The dual-mode (mock/production) architecture is highly effective for developing UXP plugins, as it allows for logic testing without needing Premiere Pro to be running.
3.  **Documentation is Key:** The official Adobe UXP documentation is the primary source of truth, but it requires careful reading to understand the nuances of the API (e.g., synchronous vs. asynchronous calls, Action-based vs. direct methods).
